# React / Next.js 웹 프론트엔드 신입 JD 리서치

작성일: 2026-04-08  
대상: React / Next.js 기반 웹 프론트엔드 신입·주니어 포지션  
작성 목적: 현재 채용 공고 기준으로 `신입에게 실제로 요구되는 역량`을 정리하고, 포트폴리오에서 무엇을 증명해야 하는지 판단하기 위함

---

## 1. 조사 기준

이번 문서는 아래 기준으로 정리했다.

- 신입·주니어 공고를 우선 참고
- 신입 공고 수가 적거나 표현이 간단한 경우, `신입 가능 / 5년 이하` 공고를 보조 자료로 사용
- 기술 그 자체보다, 채용 공고에서 반복적으로 등장하는 `역량 표현`을 중심으로 공통분모를 추출

주의:
- 아래 결론은 여러 공고를 종합한 `공통 패턴`이다.
- 회사마다 기술 스택은 다르지만, 신입에게 기대하는 기본기는 꽤 비슷하다.

---

## 2. 한 줄 결론

현재 React / Next.js 웹 프론트엔드 신입 채용에서 기업이 가장 먼저 보는 것은 `React/Next.js를 쓸 줄 아는가` 자체보다,  
`HTML/CSS/JavaScript/TypeScript 기본기`, `브라우저와 HTTP 이해`, `React 기반 화면 구현 경험`, `협업 가능한 태도`, `문제를 정의하고 끝까지 해결한 프로젝트 경험`이다.

Next.js는 중요하지만, 신입에게는 보통 `필수 기본기 위에 얹히는 프레임워크 이해`로 보는 경우가 많다.  
반대로 TypeScript, 웹 기본기, 협업 능력, 문제 해결력은 거의 공통적으로 등장한다.

---

## 3. 신입에게 요구되는 핵심 역량

### 3.1 웹 기본기

가장 자주 보이는 요구사항은 여전히 웹 기본기다.

- HTML, CSS, JavaScript를 직접 다룰 수 있는가
- 반응형 마크업을 구현할 수 있는가
- 웹 표준과 웹 접근성을 이해하는가
- 브라우저가 어떻게 동작하는지 기본 원리를 아는가
- HTTP/REST 같은 서버 통신 기초를 이해하는가

정리:
- React나 Next.js는 결국 웹 위에서 동작한다.
- 그래서 신입일수록 프레임워크 API 암기보다 `웹이 어떻게 동작하는지`를 보는 경우가 많다.

포트폴리오에서 증명하는 방법:
- 반응형 레이아웃
- 접근성을 고려한 폼/버튼/키보드 조작
- API 연동 시 요청/응답 흐름 설명
- 페이지 렌더링/라우팅/상태 흐름을 글로 설명

---

### 3.2 TypeScript 기본기

TypeScript는 거의 기본값에 가깝다.

- 단순히 "써봤다"보다 타입을 활용해 상태, API 응답, 폼 데이터, 컴포넌트 props를 안정적으로 관리할 수 있는지 본다.
- Wanted 리포트에서도 TypeScript는 프론트엔드 채용 공고에서 꾸준히 언급 빈도가 증가한 핵심 키워드로 나타난다.

정리:
- 지금 시장에서는 `React + TypeScript`가 사실상 기본 조합이다.
- 신입이라도 포트폴리오에서 타입 설계를 어느 정도 보여주는 편이 유리하다.

포트폴리오에서 증명하는 방법:
- API 응답 타입 정의
- 폼 값 검증과 타입 연결
- 컴포넌트 props 타입 설계
- `any`를 최소화한 구조 설명

---

### 3.3 React / Next.js 기본 이해

공고에서 말하는 "React/Next.js 경험"은 보통 아래 수준을 뜻한다.

- React 컴포넌트 구조를 이해하고 화면을 구현할 수 있는가
- 상태 관리가 왜 필요한지 알고 적절히 분리할 수 있는가
- Next.js 라우팅, 페이지 구성, 데이터 연동 방식을 이해하는가
- 서버 API와 프론트 화면을 자연스럽게 연결할 수 있는가

정리:
- 신입에게 SSR, RSC, 캐싱 전략까지 깊게 요구하는 경우는 상대적으로 적다.
- 하지만 `React/Next.js로 실제 프로젝트를 완성해본 경험`은 강하게 본다.

포트폴리오에서 증명하는 방법:
- React / Next.js로 만든 실제 서비스 화면
- 목록 / 상세 / 생성·수정 / 로그인 등 여러 패턴 구현 경험
- 상태 관리 선택 이유 설명
- API와 연결된 동작 흐름 설명

---

### 3.4 협업 기본기

신입 공고에서도 협업 역량은 반복적으로 나온다.

- Git 사용 경험
- Git Flow 또는 기본 브랜치 전략 이해
- 디자이너, 기획자, 백엔드와의 커뮤니케이션
- 코드 리뷰 참여 가능 여부
- 협업 툴(Slack, Jira, Notion, Figma 등) 적응력

정리:
- 신입이라도 혼자만 코드를 짜는 사람보다, 팀 안에서 소통 가능한 사람을 선호한다.
- 특히 프론트엔드는 기획/디자인/백엔드와 맞닿아 있기 때문에 커뮤니케이션이 매우 자주 언급된다.

포트폴리오에서 증명하는 방법:
- PR / 이슈 / 문서화 경험
- 요구사항을 문서로 정리한 경험
- 디자이너/기획자/백엔드와 협업한 사례
- 코드 리뷰나 피드백 반영 경험

---

### 3.5 문제 해결력과 주도성

좋은 신입 공고는 단순 구현자보다 `문제를 정의하고 해결하는 사람`을 찾는다.

- 주어진 요구사항을 그대로 구현하는 데서 끝나지 않고
- 어디가 불편한지, 어떤 구조가 위험한지, 어떻게 개선할 수 있는지를 설명할 수 있어야 한다

정리:
- "무엇을 만들었다"보다 "왜 그렇게 만들었는가"가 더 중요하다.
- 포트폴리오에서도 기능 나열보다 문제 해결 서사가 강할수록 평가가 좋아진다.

포트폴리오에서 증명하는 방법:
- AS-IS / TO-BE 구조로 정리
- 성능, UX, 데이터 정합성, 운영 편의성 등 개선 포인트 설명
- 리스크를 어떻게 줄였는지 서술

## 4. 있으면 강한 우대 역량

아래 역량은 신입 필수까지는 아니지만, 보이면 확실히 경쟁력이 올라간다.

### 4.1 성능 최적화

- 복잡한 UI를 매끄럽게 동작시키기 위한 성능 최적화 경험
- 렌더링 구조 개선
- 불필요한 요청/렌더링 감소

포트폴리오 예시:
- 이미지 최적화
- 렌더링 병목 해결
- 목록/필터 UI 개선

### 4.2 테스트 경험

- 단위 테스트
- E2E 테스트
- 테스트 환경 구축 경험

정리:
- 신입 필수는 아니지만, 테스트가 있는 포트폴리오는 확실히 차별화된다.

### 4.3 디자인 시스템 / Storybook

- 디자인 시스템 이해
- 재사용 가능한 컴포넌트 설계
- Storybook 같은 도구 경험

정리:
- 특히 Next.js + React 조직에서는 재사용성과 화면 일관성을 중요하게 보는 곳이 많다.

### 4.4 웹 접근성 / SEO / 반응형

- 웹 표준
- 웹 접근성
- SEO 이해
- 반응형 구현 경험

정리:
- 이 항목은 "있으면 좋음"이 아니라 사실상 기본기와 우대 사이 어딘가에 있다.
- 특히 서비스 회사일수록 신입에게도 꽤 자주 묻는다.

### 4.5 배포 자동화 / CI/CD / 운영 감각

- GitHub Actions 같은 CI/CD 도구 경험
- 배포 파이프라인에 대한 이해
- 운영 환경 기준 이슈 대응 경험

정리:
- 프론트엔드 신입의 필수는 아니지만, 있으면 실무 적응력이 높다고 평가받기 쉽다.

## 5. 신입 포트폴리오에서 반드시 보여줘야 할 것

채용 공고를 종합하면, 신입 포트폴리오는 아래를 보여줄 때 설득력이 높다.

### 필수

- React / Next.js 기반 실제 프로젝트 1개 이상
- HTML / CSS / JavaScript / TypeScript 기본기
- API 연동 경험
- GitHub 저장소 또는 프로젝트 링크
- 문제 해결 과정을 설명한 문서형 포트폴리오

### 있으면 좋음

- 반응형 / 접근성 고려
- 로그인 / 인증 흐름
- 상태 관리 이유 설명
- 성능 최적화나 구조 개선 사례
- 테스트 또는 검증 자동화

### 매우 좋음

- 디자인 시스템 / 공통 컴포넌트 설계
- 운영용 백오피스 또는 실제 서비스 운영 경험
- 문서화 경험
- 협업 방식과 피드백 반영 과정 설명

## 6. 현재 시장 기준 체크리스트

아래 항목에 `예 / 아니오`로 체크해보면, 신입 포트폴리오 경쟁력을 대략 점검할 수 있다.

### 기본기

- HTML / CSS / JavaScript를 직접 설명할 수 있다
- TypeScript로 컴포넌트와 API 타입을 작성할 수 있다
- 브라우저 동작 원리와 HTTP 요청 흐름을 설명할 수 있다
- React 컴포넌트와 상태 흐름을 설명할 수 있다
- Next.js에서 라우팅과 데이터 연동 구조를 설명할 수 있다

### 구현 경험

- 실제 서비스형 UI를 구현한 프로젝트가 있다
- 로그인 / 폼 / 목록 / 상세 같은 대표 패턴을 구현해봤다
- API와 연결된 화면을 만들어봤다
- 반응형 / 접근성 / SEO 중 1개 이상을 고려한 경험이 있다

### 협업 / 실무 감각

- Git / GitHub를 활용해본 경험이 있다
- 문서화 또는 코드 리뷰 경험이 있다
- 기획/디자인/백엔드와의 협업 경험을 설명할 수 있다
- 문제를 정의하고 개선한 사례를 말할 수 있다

### 차별화

- 성능 최적화 경험이 있다
- 테스트 경험이 있다
- Storybook / 디자인 시스템 경험이 있다
- GitHub Actions 등 CI/CD 경험이 있다

## 7. SteelArt 프로젝트와 연결하면 좋은 포인트

현재 저장소 기준으로 SteelArt 프로젝트는 아래 역량을 보여주기에 유리하다.

### 잘 맞는 항목

- React / Next.js 기반 실제 서비스형 관리자 UI
- 목록 / 상세 / 생성·수정 / 로그인 / 보호 경로 구현
- TypeScript 기반 프론트엔드 / 백엔드 통합 프로젝트
- API 연동, S3 업로드, Kakao Maps 연동
- 운영용 백오피스 UX 설계
- 문서화 및 AI 협업 프로세스

### 추가로 강조하면 좋은 항목

- 프론트엔드 관점에서 왜 Tailwind CSS + shadcn/ui를 선택했는지
- 반복 화면을 빠르게 구현하기 위한 UI 패턴 설계
- 로그인 리다이렉트 UX 개선
- 운영자 입력 UX 단순화
- 이미지 metadata 유실 방지 같은 문제 해결 사례

### 상대적으로 약한 항목

- 테스트 코드 자체는 강한 포트폴리오 포인트로 보이기엔 아직 약함
- Storybook / 디자인 시스템 레벨의 분리된 사례는 비교적 약함
- 성능 최적화 사례는 있다면 더 보강해서 분리해 쓰는 편이 좋음

## 8. 최종 판단

현재 Next.js / React 웹 프론트엔드 신입 채용에서 가장 중요한 것은 아래 순서다.

1. 웹 기본기
2. TypeScript
3. React / Next.js로 실제 프로젝트를 끝까지 구현한 경험
4. Git / 협업 / 커뮤니케이션
5. 문제 해결 서사

그리고 아래는 차별화 요소다.

1. 성능 최적화
2. 테스트
3. 디자인 시스템 / Storybook
4. CI/CD / 운영 감각

즉, 신입 포트폴리오는 `최신 기술 많이 썼다`보다  
`기본기가 있고, React/Next.js 프로젝트를 완성했고, 문제를 정의하고 해결할 수 있다`를 보여주는 쪽이 더 강하다.
