# SteelArt 관리자 대시보드 포트폴리오 초안

작성 목적:
- 사용자가 공유한 Figma 포트폴리오 예시처럼, `프로젝트 개요 1장 + 상세 문제해결 페이지 여러 장` 구조로 바로 옮길 수 있는 원고 초안
- 문장 톤은 "무엇을 만들었는지"보다 "왜 이렇게 설계했고 어떤 운영 문제를 해결했는지"에 초점

예시 포맷에서 가져온 핵심 구조:
1. 프로젝트 대표 페이지
2. 핵심 문제를 페이지 단위로 분리
3. 각 상세 페이지는 `문제 배경 -> 해결 방식 -> 결과/배운 점` 흐름
4. 긴 기능 나열보다 `운영상 중요했던 의사결정`을 중심으로 정리

권장 페이지 수:
- 최소 4페이지
- 권장 5페이지

## 페이지 1. 프로젝트 대표 소개

### 상단 헤더
- 프로젝트명: SteelArt 관리자 대시보드
- 한 줄 소개: 모바일 서비스 운영을 위해 사용자, 작가, 작품, 코스, 홈 배너를 관리하는 관리자 백오피스를 구축한 프로젝트
- 기간: 2026.02 ~ 2026.03
- 형태: 개인 프로젝트 또는 AI 협업 기반 관리자 시스템 구축 사례

### 링크 영역
- GitHub: 저장소 링크
- 배포: 운영 또는 데모 링크
- 문서: README 또는 ERD/시연 링크

### 대표 이미지 추천
- `docs/readme/assets/readme-cover.png`
- 또는 `docs/readme/assets/artwork-management.png`

### 개요
SteelArt 모바일 앱 운영을 위해 사용자, 작가, 작품, 코스, 홈 배너를 관리하는 관리자 백오피스입니다.  
서비스 초기부터 운영 데이터를 안정적으로 관리할 수 있도록 설계했으며, 실제 운영자가 빠르고 정확하게 업무를 처리할 수 있는 기능을 중심으로 구축했습니다.

### 내가 맡은 역할
- 풀스택 엔지니어로서 SteelArt 관리자 백오피스의 도메인 구조를 설계하고, 사용자·작가·작품·코스·홈 배너 관리 기능을 구현했습니다.
- 서버 엔지니어 관점에서 관리자용 API를 설계·개발하고, MySQL·AWS S3·Kakao Maps 연동을 포함한 운영 데이터 처리 구조를 구축했습니다.
- AI와 협업하며 조사용 마크다운 파일과 구현 계획용 마크다운 파일을 작성하고, 구현 전 사전 피드백과 반복 검증을 통해 결과물의 완성도와 데이터 정합성을 높였습니다.

### 기술 스택
- Frontend: Next.js, TypeScript, Tailwind CSS, shadcn/ui
- Backend: MySQL, NextAuth
- Infra/External: AWS S3, Kakao Maps

### JD 기준으로 강조할 역량
- React / Next.js로 실제 서비스형 관리자 화면을 끝까지 구현한 경험
- TypeScript 기반으로 폼, 상태, API 흐름을 안정적으로 연결한 경험
- 기능 나열보다 문제 해결 중심으로 프론트엔드 구조를 설계한 경험
- 운영자 관점에서 입력 흐름과 인증 흐름을 개선한 UI/UX 설계 경험
- 마크다운 문서와 반복 피드백을 기반으로 협업 품질을 끌어올린 경험

### 하단 한 줄 요약
실제 서비스형 관리자 화면을 완성하고, TypeScript 기반 화면 연결, 운영자 UX 개선, 문서 기반 협업까지 함께 보여줄 수 있는 프로젝트입니다.

## 페이지 2. Frontend 개발

### 요약
Next.js와 TypeScript를 기반으로 목록, 상세, 생성·수정, 로그인 같은 관리자 핵심 화면을 실제 서비스형 UI로 완성했습니다.  
이 과정에서 단순히 화면을 많이 만드는 것보다, 디자인이 정해지지 않은 상황에서 빠르게 반복 가능한 구조를 만들고 운영자가 실제로 쓰기 쉬운 입력 흐름과 인증 흐름을 설계하는 데 집중했습니다.

### 내용 1 제목
디자인이 확정되지 않은 상태에서 빠르게 관리자 UI를 구축할 수 있는 프론트엔드 구조 설계

#### AS-IS
- 관리자 페이지 기획은 있었지만, 구체적인 디자인 시스템은 정해지지 않은 상태였습니다.
- 목록, 상세, 폼처럼 반복되는 화면이 많아 매번 커스텀 CSS로 구현하면 속도와 일관성 모두 떨어질 수 있었습니다.

#### TO-BE
- Tailwind CSS와 shadcn/ui를 선택해 공통 UI 패턴을 빠르게 조합할 수 있는 구조를 만들었습니다.
- 특히 AI가 이미 잘 학습한 스타일 체계를 활용해 초안 생성 속도를 높이고, 이후에도 같은 컴포넌트 규칙 안에서 화면을 수정·보완할 수 있도록 구성했습니다.

### 내용 2 제목
관리자 인증과 보호 경로 UX 개선

#### AS-IS
- 비로그인 상태에서 보호 페이지에 접근하면 맥락 없이 로그인 페이지로 이동해 운영 흐름이 끊길 수 있었습니다.
- 운영 배포 환경에서는 로그인 성공 직후 다시 로그인 페이지로 튕기는 레이스 컨디션이 발생했습니다.

#### TO-BE
- middleware로 `/admin/*`, `/api/admin/*` 경로를 보호하고, 비로그인 접근 시 `reason`과 `next`를 함께 넘겨 로그인 후 원래 경로로 복귀하도록 구성했습니다.
- 로그인 중에는 전면 로딩 오버레이를 표시하고, 프로덕션 레이스 컨디션은 `window.location.replace(...)`로 해결해 관리자 흐름이 끊기지 않도록 개선했습니다.

### 내용 3 제목
작품 등록 화면에서 장소 입력과 지도 UX 통합

#### AS-IS
- 장소를 별도 메뉴에서 관리하면 작품 등록 흐름이 끊기고 운영자가 여러 화면을 오가야 했습니다.
- 주소와 좌표를 모두 수동 입력하면 입력 비용이 커지고 실수 가능성이 높았습니다.

#### TO-BE
- 작품 생성/수정 폼 안에 장소 정보를 통합하고, Kakao geocoding으로 주소 기반 좌표 자동 입력, 수동 lat/lng 수정, 지도 미리보기와 마커 표시를 지원했습니다.
- 운영자는 한 화면에서 작품과 장소 정보를 함께 처리할 수 있게 되었고, 작품·장소·업로드 결과를 하나의 폼 흐름 안에서 안정적으로 다룰 수 있게 됐습니다.

## 페이지 3. Backend 개발

### 요약
이 프로젝트의 백엔드는 단순 CRUD API를 만드는 것이 아니라, 프론트엔드 화면이 안정적으로 동작할 수 있도록 운영 정책과 데이터 계약을 먼저 구조화하는 데 초점을 맞췄습니다.  
ERD 설계, DB 계약 정리, 파라미터 바인딩, connection pool, transaction 관리, 업로드와 메타데이터 정합성까지 운영 환경에서 실제 문제가 될 수 있는 요소를 먼저 정리했습니다.

### 내용 1 제목
운영 정책을 담는 ERD와 데이터 모델 설계

#### AS-IS
- 운영 도메인을 명확히 나누지 않으면 삭제/복구 정책이 뒤섞이고, 기능이 늘어날수록 데이터 책임이 불분명해질 수 있었습니다.
- 작품, 장소, 코스, 체크인처럼 참조 관계가 있는 데이터는 단순 CRUD만으로 다루기 어려웠습니다.

#### TO-BE
- `artists`, `artworks`, `artwork_images`, `artwork_festivals`, `courses`, `course_items`, `course_checkins`, `home_banners`, `places`, `zones` 중심으로 관계를 정리했습니다.
- `artists/artworks/courses/places`는 soft delete, `home_banners`는 hard delete로 정책을 구분하고, `course_items`는 체크인 존재 시 삭제를 차단하도록 설계했습니다.

### 내용 2 제목
SQL injection 방지와 connection/transaction 관리

#### AS-IS
- SQL 문자열을 직접 조합하거나 요청마다 연결을 새로 열면 API가 늘어날수록 안전성과 운영성이 낮아질 수 있었습니다.
- 작품 저장, 장소 저장, 이미지 저장처럼 여러 쓰기 작업이 한 번에 이뤄지는 구간은 중간 실패 시 데이터 불일치가 생길 수 있었습니다.

#### TO-BE
- `mysql2` 기반 connection pool과 SSL 자동 처리 구조를 구성하고, 모든 쓰기 API에 `?` 파라미터 바인딩을 적용해 SQL injection 가능성을 낮췄습니다.
- `withTransaction` wrapper를 사용해 place 생성/수정, artwork 저장, image/festival 저장처럼 하나의 작업 단위가 트랜잭션으로 처리되도록 구현했습니다.

### 내용 3 제목
S3 업로드와 이미지 메타데이터 정합성 보장

#### AS-IS
- 작품 이미지는 업로드만 되더라도 `image_width`, `image_height`가 저장되지 않으면 수정 시 `DELETE -> INSERT` 과정에서 metadata가 유실될 수 있었습니다.
- 수동 URL 입력 경로에서는 잘못된 이미지가 들어와도 저장될 가능성이 있었습니다.

#### TO-BE
- S3 Presigned URL 기반 업로드 API를 구현하고, 이미지 파일과 수동 URL 모두에서 width/height를 추출하도록 연결했습니다.
- 작품 저장 시 이미지 metadata가 없으면 submit을 막고, 기존 운영 데이터 221건에 대해서는 backfill 스크립트를 작성해 메타데이터를 정리했습니다.

### 내용 4 제목
코스/배너 운영 안전장치와 공통 API 응답 구조

#### AS-IS
- 코스 아이템 순서 변경, 체크인 이력, 홈 배너 노출 순서 같은 데이터는 잘못 수정되면 사용자 경험에 직접 영향을 줄 수 있었습니다.
- API마다 에러 형식이 다르면 관리자 화면에서 실패 원인을 일관되게 처리하기 어려웠습니다.

#### TO-BE
- 코스 아이템 reorder 시 기존 `id`는 유지하고 `seq`만 갱신하도록 했고, `seq`는 1부터 연속값이 되도록 강제했습니다.
- 체크인 데이터가 있는 코스 아이템은 `409 CHECKIN_EXISTS`로 삭제를 차단했고, 홈 배너는 전체 배너를 포함한 reorder만 허용하도록 검증했습니다.
- 공통 API 응답 형식과 에러 코드를 사용해 관리자 화면에서 실패 원인을 일관되게 처리할 수 있도록 구성했습니다.

## 페이지 4. 검증 및 AI 협업 방식

### 요약
이 프로젝트에서는 AI를 바로 코드 생성기로 쓰기보다, 조사와 설계 문서를 함께 정리하는 협업 도구로 활용했습니다.  
구현 전에 마크다운 문서로 조사 범위와 구현 계획을 먼저 정리하고, 사람이 예상 가능한 문제를 선제적으로 피드백한 뒤 구현과 검증을 반복하는 방식으로 결과물의 품질과 협업 완성도를 높였습니다.

### 내용 1 제목
실제 운영 기준으로 검증하는 품질 관리

#### AS-IS
- 코드가 컴파일되고 화면이 보이는 것만으로는 관리자 시스템 품질을 보장할 수 없었습니다.
- 인증, MySQL, S3 같은 실제 연동 구간은 로컬 구현만으로는 운영 적합성을 판단하기 어려웠습니다.

#### TO-BE
- `pnpm exec tsc --noEmit`, `pnpm lint`, `pnpm build`를 기본 검증으로 사용하고, Playwright 기반 headed 검증까지 수행했습니다.
- 실제 MySQL, 실제 S3, 실제 인증 흐름 기준으로 동작을 확인하면서 운영 환경에서의 안정성을 검증했습니다.

### 내용 2 제목
마크다운 문서 기반 AI 협업 프로세스 설계

#### AS-IS
- AI에게 바로 코드 생성을 요청하면 과설계, 잘못된 데이터 흐름, 운영 정책 누락이 발생할 수 있었습니다.
- 관리자 시스템은 스펙과 데이터 정책이 명확하지 않으면 재작업 비용이 크게 늘어날 수 있었습니다.

#### TO-BE
- 먼저 조사용 마크다운 파일에서 DB 구조, 운영 정책, API 계약, 인프라 제약을 정리하고, 구현 계획용 마크다운 파일에서 구현 순서와 리스크를 구체화했습니다.
- 그 뒤 내가 AI 결과물을 미리 예상해 사전 피드백을 주고, 이후 구현과 재검증을 반복하는 방식으로 결과물의 완성도와 데이터 정합성을 높였습니다.

## Figma로 옮길 때의 편집 가이드

### 톤
- 한 페이지에 메시지는 하나만 남기기
- "기능 목록"보다 "문제를 어떻게 풀었는지"가 보이게 쓰기
- 수치가 있으면 짧게 강조하기

### 시각 구성
- 1페이지: 대표 이미지 + 개요/역할/스택
- 2페이지 이후: 큰 제목 + 2~3개 문단 + 캡처 1장 또는 다이어그램 1장
- `AS-IS / TO-BE`는 긴 문장보다 짧은 문단 2개로 정리

### 문장 압축 원칙
- 한 문단은 3문장 이내
- 기술명은 나열보다 역할과 함께 표기
- "구현했다"보다 "왜 그렇게 했는지"를 먼저 적기

### 추천 순서
1. 페이지 1 그대로 작성
2. 페이지 2에서 프론트엔드 구현력과 사용자 경험 설계 역량을 먼저 보여주기
3. 페이지 3에서 백엔드 설계와 운영 안정성 역량 보강
4. 페이지 4에서 검증 방식과 AI 협업 프로세스로 작업 방식 차별점 마무리
